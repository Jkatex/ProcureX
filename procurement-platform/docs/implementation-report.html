1. Abstract

ProcureX is a modular procurement platform designed to support the full lifecycle of public and private procurement operations. The system includes public marketplace discovery, account registration, company verification, tender planning, tender publication, supplier bidding, bid evaluation, award grouping, award recommendation, contract negotiation, post-award tracking, communication, records management, support operations, and platform compliance administration.

The implementation uses a production monorepo named procurement-platform. The frontend is implemented with React, Vite, Material UI, Redux Toolkit, React Router, i18next, and supporting shared UI utilities. The backend is implemented with TypeScript, Express, Helmet, Prisma, modular API boundaries, PostgreSQL multi-schema modeling, and row-level security foundations. The original procurex-ui prototype is retained as the visual and workflow reference, while the production React client owns the operational implementation.

Academically, the system demonstrates a layered information-system architecture: presentation concerns are isolated in the React client, process orchestration is concentrated in domain services, persistence is represented through Prisma and PostgreSQL schemas, and governance is supported through audit, records, compliance, and row-level access controls. This report therefore evaluates not only what was implemented, but also the architectural quality, data integrity model, validation strategy, and future production-readiness path.

Documentation note: This report is a documentation deliverable only. It does not introduce public API changes, database schema changes, or application behavior changes.

2. Introduction

Procurement systems require strong traceability, controlled access, structured workflows, and clear collaboration between buyers, suppliers, evaluators, approvers, and administrators. ProcureX addresses these needs by organizing procurement activity around companies and workflow assignments rather than treating buyer and supplier as simple login roles.

The system separates presentation reference from production execution. The prototype in ../procurex-ui captures the intended user experience, while procurement-platform/client implements the production interface. This separation allows the project to preserve the design intent while progressively adding authentication, routing, state management, localization, API integration, tests, and database enforcement.

3. Objectives

The implementation objectives were defined around system completeness, maintainability, and future production readiness.

  1. Implement a production workspace that reflects the ProcureX reference user interface.
  2. Provide a modular backend where each procurement domain has a clear API boundary.
  3. Model identity, organizations, buyer and supplier capabilities, tenders, bids, evaluation, awards, contracts, records, and compliance data.
  4. Separate account type from procurement capability so a company can act as buyer, supplier, or both.
  5. Prepare row-level security foundations for organization-aware data access.
  6. Support screenshot-based UI parity validation between the reference prototype and the production React client.
  7. Provide tests and scripts that support local development, validation, database setup, and future enhancement.

4. Methodology

The development approach used a reference-driven migration methodology. The original static ProcureX interface remained the product and visual source of truth. Production implementation then translated that reference into React pages, guarded routes, domain modules, database models, and repeatable tests.

4.1 Reference UI Method

UI parity is maintained by capturing screenshots from both the static prototype and the production React target. This makes design drift visible and gives the team concrete evidence for report, review, and validation work.

4.2 Modular Development Method

Backend modules follow a consistent structure: 

routes.ts -> controller.ts -> service.ts -> repository.ts -> validators.ts -> types.ts

This pattern keeps HTTP transport, business behavior, persistence, validation, and contracts separated. It also gives future developers a predictable place to add new behavior.

5. System Architecture

ProcureX is implemented as a monorepo with separate workspaces for the frontend client, backend server, and shared contracts. The architecture supports independent growth of user interface features, API modules, database models, and future intelligence services.

5.1 Architectural Principles

  • The reference UI is not served directly by production.
  • The React client owns production user interaction and route behavior.
  • The Express server owns API routing, module registration, and backend orchestration.
  • The Prisma schema is the source of database model truth.
  • Organization-aware security is enforced through identity rules and RLS foundations.

7. Backend Implementation

The backend is a TypeScript Express application that provides the controlled service boundary between the React client and the PostgreSQL data layer. It centralizes HTTP app creation, security middleware, module registration, JSON parsing, CORS configuration, health reporting, 404 handling, and centralized error responses. This makes the server a disciplined application-service layer rather than a collection of disconnected route handlers.

The backend entrypoint is organized around createApp() in server/src/app.ts. The application validates production security configuration, sets proxy trust, applies Helmet security headers, configures a content security policy, restricts CORS origins, limits JSON bodies to 2 MB, exposes /health, mounts domain routers from the module registry, and finally applies not-found and error middleware. This sequence is important because cross-cutting security and parsing behavior are established before any domain module receives a request.

7.1 Express Bootstrap And HTTP Boundary

The bootstrap layer implements the outermost boundary of the system. Its responsibility is not procurement logic, but safe and predictable HTTP behavior. Helmet is configured with a content security policy that limits script, image, style, font, frame, connection, base URI, and frame ancestor sources. CORS is resolved from configured allowed origins, with local origins available for development. Requests that do not match an approved origin are rejected before they can reach module code.

Backend Concern Implementation Architectural Value
Security headers helmet() with explicit CSP directives. Reduces browser-side attack surface and documents accepted resource origins.
CORS Origin callback checks configured client origins and local development origins. Prevents arbitrary browser origins from invoking protected APIs.
Request body size express.json({ limit: '2mb' }). Limits accidental or hostile oversized JSON payloads.
Health reporting GET /health returns service status and registered modules. Supports operational checks and automated assertions in tests.
Error shape Centralized 404 and error middleware. Keeps client-facing failure responses consistent across modules.

7.2 Module Pattern

Each module uses the same implementation pattern so behavior remains consistent across procurement domains. The pattern is academically significant because it separates transport, validation, orchestration, persistence, and type contracts. This lowers coupling and makes the system easier to reason about during testing, review, and future extension.

File Responsibility
routes.ts Defines HTTP endpoints and maps them to controller handlers. Route files should remain thin and should not contain business decisions.
controller.ts Handles request and response concerns, including parameter extraction, status codes, and response formatting.
service.ts Owns business behavior, workflow orchestration, permission-sensitive decisions, and coordination between repositories or providers.
repository.ts Owns persistence and data access through Prisma or provider-specific integrations.
validators.ts Owns request validation, usually with Zod schemas, so invalid input is rejected before service behavior runs.
types.ts Owns module-local contracts, DTOs, enums, and metadata used by the module implementation and tests.

A typical request therefore travels through a clear path: 

HTTP request
  -> Express middleware
  -> module route
  -> controller
  -> validator
  -> service
  -> repository
  -> Prisma/PostgreSQL
  -> response or centralized error handler

This request lifecycle supports maintainability because a change in one concern has an obvious home. For example, a new validation rule belongs in validators.ts, a procurement workflow rule belongs in service.ts, and a query optimization belongs in repository.ts.

8.3 Core Data Rules

  • Accounts are only USER or ADMIN.
  • A normal user belongs to a company through OrganizationMember.
  • A company can act as buyer, supplier, or both through OrganizationCapability.
  • Tender ownership uses buyerOrgId.
  • Bid ownership uses both buyerOrgId and supplierOrgId.
  • Evaluation permissions use workflow assignments and RLS-aware access.

These rules prevent a common procurement-system design flaw: treating buyer, supplier, evaluator, and approver as permanent login categories. ProcureX instead treats them as contextual capabilities and workflow responsibilities. This provides a better foundation for organizations that have multiple departments, multiple users, or dual buyer-supplier participation.

8.6 Product Security Model

The most important security decision is that buyer and supplier are not account types. They are organization capabilities. This makes it possible for the same company to buy, supply, or do both while still keeping platform administrators separate from procurement operators.

This security model also clarifies administrative authority. An ADMIN account is a platform compliance account, not a buyer or supplier. Admin access is therefore treated as platform oversight, while procurement actions are attached to organizations and workflow records. This distinction helps preserve accountability because business actions and oversight actions can be audited separately.

8.7.2 Identity Schema

The identity schema models users, accounts, preferences, policy acceptances, sessions, MFA factors, identity challenges, verification profiles, verification profile history, digital signatures, signing credentials, verification documents, and screening checks. These models exist because procurement access depends on knowing who the user is, whether the account is active, whether the user's organization has been verified, and whether signing or verification evidence exists.

The identity schema connects to organization data through verification profiles and organization membership. It supports frontend registration, sign-in, verification status, profile, and admin review features. The result is a strong identity foundation for procurement workflows. Testing evidence includes identity production-auth tests, delivery-provider tests, signing tests, module validators, seed execution, and RLS policies for users, accounts, sessions, and verification profiles.

8.7.3 Organization Schema

The organization schema models organizations, members, capabilities, organization profiles, buyer profiles, supplier profiles, trust tier history, and permission overrides. It exists to represent companies and institutional participation rather than treating procurement behavior as a simple user role. A normal account can belong to an organization, and that organization can act as buyer, supplier, or both.

This schema connects identity to procurement by giving tenders a buyer organization and bids a supplier organization. It also supports route guards, RLS context, profile review, trust/risk display, and future permission expansion. Validation is supported by Prisma validation, identity verification tests, seed data, RLS verification, and module-validator checks.

8.7.4 Procurement Schema

The procurement schema models tenders, tender amendments, saved tenders, procurement plans, plan lines, tender categories, tender documents, tender requirements, tender milestones, and tender commercial items. Procurement models connect to organization through buyerOrgId and connect to later bidding, evaluation, award, and contract records. The result is a buyer-owned tender data model that can support marketplace visibility and supplier review. Testing includes Prisma validation, procurement repository tests, service tests, controller tests, marketplace seed tests, award-ready seed tests, and migration checks for tender amendments and saved tenders.

8.7.5 Bidding Schema

The bidding schema models bids, bid versions, bid documents, bid responses, bid receipts, and bid samples. Bidding connects tenders, buyer organizations, and supplier organizations. The data model supports active-bid uniqueness, supplier ownership, buyer visibility, administrative envelopes, and sample tracking. The result is a structured response model that can feed evaluation. Testing evidence includes bidding tests, procurement API tests, active-bid uniqueness migration validation, marketplace demo seed validation, and client bidding workspace tests.

8.7.6 Evaluation Schema

The evaluation schema models evaluation workspaces, workflow assignments, evaluation criteria, and evaluation scores. It exists to turn submitted bids into structured review evidence. 8.7.17 RLS And Tenant Isolation

Row-level security was implemented through PostgreSQL helper functions and policies. The application sets context variables inside withDbContext(): app.current_user_id, app.current_organization_id, app.current_account_type, and app.current_capabilities. SQL helper functions read those values as app.current_user_id(), app.current_organization_id(), app.current_account_type(), and app.current_capabilities().

Policies then use checks such as app.is_admin(), app.same_org(...), and app.has_capability(...). This means access can be enforced by the database, not only by frontend screens or service filters. The result is tenant isolation for organization-owned rows and self-access policies for identity records. Validation includes verify:rls, database validation scripts, policy review, and service tests that depend on correct organization context.

8.7.18 Migrations And Seed-Data Validation

Migrations were implemented as incremental Prisma migration directories. They capture schema evolution from initialization through identity, public content, signatures, IAM access, procurement planning, awards, support, compliance rules, admin datastore, bidding uniqueness, saved tenders, dual capabilities, tender amendments, administrative envelopes, bid samples, and award groups.

Seed scripts were implemented to create repeatable development and demo states. The base seed prepares core data, while optional seeds prepare marketplace tenders, award-ready evaluation data, award-contract demo workflows, and evaluation intake scenarios. db:seed:twice validates idempotency by running the seed process more than once. The result is a database that can be rebuilt, demonstrated, and tested consistently. Validation includes db:validate, db:migrate, db:seed, db:seed:twice, seed-specific tests, repository tests, and service tests.

The main database result is a normalized, auditable model of the procurement lifecycle. Instead of only storing high-level tender and bid records, the database stores the supporting evidence required for verification, evaluation, award approval, contract execution, payment, compliance, communication, and records review. This makes the database a governance asset, not merely an application storage layer.

9. Implementation And Technical Quality

Technical quality is evaluated by examining how the implementation handles separation of concerns, maintainability, security, traceability, scalability, and known limitations. ProcureX is not only a collection of screens and tables; it is a procurement information system whose correctness depends on process integrity, evidence preservation, and controlled access.

9.1 Separation Of Concerns

The monorepo separates client, server, shared contracts, documentation, scripts, and future machine-learning services. Inside the server, modules separate HTTP routing from domain behavior and persistence. Inside the database, PostgreSQL schemas separate identity, organization, procurement, bidding, evaluation, contract, financial, compliance, records, and support domains. This repeated separation at repository, application, and database levels is a major architectural strength because it reduces accidental coupling.

9.2 Maintainability

Maintainability is supported by predictable folders, typed TypeScript implementation, Prisma models, Zod validation, shared contracts, and focused tests. A developer who understands one module can usually navigate another because the same structural pattern appears across the backend. The frontend uses a comparable feature-oriented structure, so procurement, bidding, evaluation, awards, communication, records, public content, and admin surfaces can evolve without every change becoming an application-wide refactor.

The main maintainability risk is breadth. The data model is already extensive, especially in awards, contracts, lifecycle management, compliance, and finance. As business logic becomes richer, the project should keep service functions focused, expand repository tests, and resist embedding cross-domain rules in UI components.

9.3 Security Posture

The current security posture includes browser security headers, restricted CORS, bounded JSON parsing, route guards on the client, identity/session APIs on the server, admin route separation, Prisma database access, and PostgreSQL RLS foundations. The most important security principle is organization-aware access: data visibility should follow the current user, organization, account type, and capability context.

Additional production hardening should focus on complete authentication token policy, secure cookie or bearer-token handling, production email and SMS provider configuration, stricter audit coverage for every decision-changing action, file upload scanning, and repeatable RLS verification in CI or deployment checks.

9.4 Traceability And Auditability

Procurement systems must explain why a decision was made, who made it, what evidence was available, and which records support the result. ProcureX addresses this through explicit models for audit events, record entries, communication items, verification history, screening checks, approval steps, award notices, contract versions, evidence, disputes, termination records, payment approvals, and compliance cases.

The academic value of this design is that records are not treated as an afterthought. They are modeled as part of the procurement lifecycle, allowing operational activity to become reviewable evidence.

9.5 Scalability Readiness

ProcureX is prepared for moderate architectural scaling because the frontend, backend, database model, and future intelligence service are separated. The backend modules can be optimized independently, Prisma repositories can be profiled per workflow, and the future ml-service can host supplier matching, risk analysis, or benchmark workloads without overloading the Express application with machine-learning concerns.

Scalability work should remain evidence-driven. The next production steps should include query profiling, index review for high-volume tables, pagination enforcement, background job boundaries for heavy document or notification work, and caching only where repeated reads have been measured.

9.6 Limitations And Technical Debt

The implementation is strong as a foundation, but several limitations remain. Some modules are more mature than others, and several backend boundaries exist ahead of complete business behavior. Migration history contains local or repeated naming that should be cleaned before production release. RLS foundations exist, but sensitive workflows should receive expanded policy coverage and automated verification. Finally, screenshot parity validates visual alignment but does not replace end-to-end workflow testing against live APIs.

13. Conclusion And Recommendations

The ProcureX implementation establishes a strong foundation for a production procurement platform. It combines a visual reference workflow, production React pages, modular Express APIs, Prisma database modeling, and organization-aware access rules. The design allows the product to preserve the experience of the original prototype while moving toward a maintainable and secure application architecture.

13.1 Conclusion

The implementation successfully models the major procurement lifecycle areas: public discovery, onboarding, verification, procurement planning, tender management, bidding, evaluation, awards, contracts, communication, records, and administration. The separation of account type from organization capability is especially important because it supports realistic procurement operations where companies may act as buyers, suppliers, or both.

From a system-design perspective, ProcureX demonstrates a coherent layered architecture. The React client owns presentation and navigation, the Express server owns API boundaries and domain orchestration, Prisma owns the typed application data model, PostgreSQL owns relational persistence and RLS enforcement, and documentation plus screenshot assets preserve the design intent. This separation is academically sound because each layer has a distinct responsibility and can be evaluated independently.

The strongest implementation qualities are modularity, traceability, and organization-aware access modeling. The most important remaining risks are production hardening, uneven module maturity, broader RLS coverage, migration hygiene, and the need for more end-to-end tests against live API-backed workflows.

13.2 Recommendations

  1. Continue converting prototype-derived screens into richer production React workflows where backend APIs are ready, while keeping page behavior aligned with actual domain services.
  2. Expand module tests as each domain moves from foundation status to full business behavior, especially for bidding, evaluation, awards, contract lifecycle, finance, and compliance workflows.
  3. Keep UI parity screenshots updated during design or layout changes, but supplement them with API-backed end-to-end tests for critical buyer, supplier, evaluator, approver, and admin journeys.
  4. Strengthen RLS verification as more sensitive procurement records become active, and require new sensitive tables to include access-policy tests before release.
  5. Review migration history before production deployment so local or duplicate migration names do not become long-term operational debt.
  6. Introduce the future intelligence service only after stable integration contracts are defined, including clear ownership for supplier matching, market benchmarking, and risk-scoring outputs.
  7. Add operational monitoring for health checks, API errors, slow database queries, failed delivery-provider calls, audit-log creation, and support-ticket volume.
  8. Document a production data-governance policy covering retention, audit review, evidence handling, file storage, administrative access, and incident response.

Overall, ProcureX is a credible foundation for a production procurement platform. It already contains the essential architectural pieces: lifecycle workflow modules, organization-aware identity modeling, PostgreSQL-backed persistence, security foundations, visual evidence, and a growing test suite. The recommended next phase is not a broad rewrite, but disciplined hardening: complete the highest-value workflow services, prove access isolation, validate live end-to-end journeys, and prepare migration and monitoring practices for production operation.

14.2 Authentication And Token Flow

Authentication uses several different token or code types, and each has a different purpose. The frontend auth slice reads a stored bearer token from client/src/shared/api/authToken.ts, keeps it in Redux auth state, and clears it on sign-out or failed hydration. The token is stored under the browser localStorage key procurex.authToken. The shared Axios client in client/src/shared/api/http.ts reads this value before each request and sends it as Authorization: Bearer <token>.

On the backend, sign-in creates an opaque random session token using Node crypto. The server returns the raw token to the client only once. The database stores a SHA-256 hash of the token as tokenHash, not the raw token. Later requests send the raw bearer token, the server hashes it again, and the identity repository searches for an active matching session. This design reduces damage if the sessions table is exposed, because stored session records do not directly contain reusable raw tokens.

Token Or Code Where It Appears Purpose Why It Is Used
Bearer session token Client localStorage, Axios authorization header, backend identity service. Authenticates API requests after sign-in. Allows stateless HTTP requests while the database keeps revocable session records.
Session token hash identity.sessions.tokenHash. Stores a non-raw representation of the session token. Prevents direct reuse of database session values as bearer tokens.
Turnstile token Registration, sign-in, forgot-password, reset-code flows. Proves a human/browser challenge was completed. Reduces automated abuse of identity endpoints.
Phone OTP code Registration start and phone verification challenges. Confirms the user controls the submitted phone number. Supports identity confidence and prevents unverified phone ownership.
Email activation code Email activation challenge after phone verification. Confirms the user controls the submitted email address. Ensures future account recovery and notifications reach the right user.
Password reset code Forgot-password and reset-password flows. Allows password recovery after email challenge. Avoids immediately revealing whether an account exists while still allowing recovery.
Digital-signature keyphrase Identity verification submission and signing credential code. Unlocks signing behavior for verification payloads. Creates signed evidence for verification consent and submitted identity data.

14.3 Authentication Feature File-By-File

The authentication feature starts on the client. The auth API helper in client/src/features/auth/api/index.ts defines the exact HTTP calls for registration start, OTP verification, OTP resend, email activation, activation resend, password setup, sign-in, forgot password, reset-code resend, reset-code verification, password reset, session lookup, access lookup, and sign-out. The auth slice in client/src/features/auth/slice.ts wraps those calls in Redux Toolkit async thunks, so UI components receive loading, success, error, token, user, and session-expired state consistently.

The server receives those requests through server/src/modules/identity/routes.ts and controller.ts. The validators file checks request body shape. The service file owns behavior: email normalization, phone normalization, OTP generation, password hashing, session token creation, challenge attempt limits, resend cooldowns, audit events, registry lookup, screening, signing, and verification submission. The repository file owns Prisma reads and writes. This is why authentication is not one file: the feature is split so UI state, HTTP transport, validation, business rules, and database persistence each have a clear home.

14.4 Turnstile Anti-Abuse Flow

Turnstile is used to protect sensitive identity entry points from automated abuse. The client sends a turnstileToken with registration, sign-in, forgot-password, reset-code, and reset-password actions. The server helper in server/src/security/turnstile.ts verifies the token. In local development, the helper allows a local-dev-turnstile: token prefix or Cloudflare's testing secret. In production, it posts the secret and response token to the Cloudflare Turnstile siteverify endpoint.

The purpose of this design is to keep anti-abuse logic outside page components and outside identity business rules. Pages only collect the token, the identity API transports it, and the security helper verifies it. This makes the implementation easier to test and safer to maintain.

14.6 Permissions, Capabilities, And Feature Gates

Permissions are computed in server/src/security/accessPolicy.ts. The code takes account type, verification status, organization capabilities, trust tier, risk level, and screening status, then returns permissions and feature gates. Admin accounts receive platform-wide permissions. Normal users receive procurement permissions through organization capability and trust/risk context. The temporary development switch currently permits core procurement permissions while trust gates are being iterated.

This explains why buyer and supplier are not stored as account types. They are organization capabilities. A user account can remain a normal USER, while the organization determines whether tender creation, bidding, evaluation, award management, contract management, signing, or tracking is available. Frontend route guards consume the session user and feature gates, while backend services call identity methods such as requireSession, requirePermission, or requireAdmin.

14.8 How To Read A File Line By Line

For any feature file, read from imports to exported functions. Imports show dependencies. Constants show policy values, such as expiry minutes, retry limits, feature-gate defaults, provider names, or API paths. Type declarations show the shape of data entering and leaving the file. Helper functions usually normalize, validate, hash, format, or map data. Exported functions, classes, and React components are the public behavior of the file. Tests show what the authors considered important enough to protect. 

Code reading order:
1. Directory name: which feature owns this file?
2. Imports: what does this file depend on?
3. Constants: what policy values does it encode?
4. Types: what data shape does it accept and return?
5. Helpers: what repeated behavior is hidden behind names?
6. Main function/class/component: what business action happens?
7. Repository/API calls: where does data go next?
8. Errors: what can fail, and how is failure reported?
9. Tests: what behavior is verified?

15.1 Engineering Handover Orientation

Purpose, reading order, and ownership boundaries for maintainers.

This chapter is written as an engineering handover layer on top of the formal report. Each plate is intentionally print-friendly: it starts on a new page, names the owned behavior, identifies the code paths or runtime contracts involved, and explains how a maintainer should reason about the implementation without changing application behavior.

The handover scope is the implemented ProcureX workspace as represented in the current repository. The frontend is the React application in client/src; the backend is the Express and Prisma application in server/src; shared account and organization contracts live in shared/src; local infrastructure is managed by Docker Compose. The static procurex-ui prototype remains a visual and workflow reference, while the production application is the React client registered through the page registry and router.

The report avoids speculative implementation claims. When a subsystem is documented here, the text describes an observable route, component, module boundary, schema model, script, test, or screenshot asset that exists in the repository.
