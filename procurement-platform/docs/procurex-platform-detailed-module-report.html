Executive Overview

ProcureX is implemented as a full-stack procurement lifecycle platform. It supports public discovery, registration, identity verification, organization capability management, tender planning, tender creation, supplier bidding, bid evaluation, award recommendation, contract formation, post-award execution, payment tracking, support, communication, records, compliance administration, and analytics-oriented intelligence boundaries.

The platform is organized as a TypeScript monorepo under procurement-platform. The production client is a React and Vite application. The backend is an Express API with Prisma persistence. The database model is PostgreSQL-oriented and split across functional schemas such as identity, organization, procurement, bidding, evaluation, contract, financial, compliance, communication, documents, records, intelligence, integration, content, and support.

The implementation separates login identity from procurement capability. A user account is either USER or ADMIN. Buyer and supplier behavior comes from organization capabilities and workflow context. This is important because a normal company account can be buyer, supplier, or both without turning the login account type into a procurement role.

Runtime support is provided through Docker Compose infrastructure: PostgreSQL for transactional storage, Redis for rate-limit backing where configured, Elasticsearch for search-oriented admin/intelligence use, and MinIO or S3-compatible storage for uploaded objects. Local scripts in scripts/ start the platform, run environment-aware commands, generate ProcureX pages, and capture UI parity screenshots.

Platform Architecture

Frontend Shape

The client application uses React 19, Vite, React Router, Redux Toolkit, Axios, MUI, i18next, Recharts, DOMPurify, ExcelJS, and PDF helpers. Routing is centralized in client/src/app/routes.tsx, with lazy page loading through procurexPageRegistry. Shared route guard components decide whether a page is public, authenticated, verified-only, or admin-only.

Backend Shape

The Express app is created in server/src/app.ts. It applies Helmet, CORS checks, JSON body limits, registered module routers, health reporting, 404 handling, and centralized error handling. Modules are registered in server/src/modules/index.ts and follow a repeated boundary: routes, controller, service, repository, validators, and types.

Database Shape

Prisma is the model and migration source. The schema includes enums for account, tender, bid, evaluation, award, contract, financial, compliance, communication, and support states. Core domain models preserve ownership through user ids, organization ids, buyer organization ids, supplier organization ids, workflow assignment ids, and document references.

Security Shape

The platform uses browser security headers, CSP directives, CORS allowlists, rate limits, Turnstile verification for public auth, bearer-token API calls, route guards, admin-only routes, validation-first controllers, standardized safe error responses, signing keyphrases for sensitive actions, audit events, and RLS-aware database context.

Registered Backend Modules

The API currently registers these base paths: /api/public, /api/identity, /api/organization, /api/procurement, /api/bidding, /api/evaluation, /api/award-contract, /api/post-award, /api/financial, /api/compliance-admin, /api/communication, /api/dashboard, /api/records, /api/intelligence, /api/integration, /api/documents, /api/support, and /api/help-centre. The app also aliases the evaluation router under /api/evaluations for client compatibility.

Typical Request Flow

React page or hook
  -> feature API helper
  -> shared Axios apiClient with bearer token and locale headers
  -> Express route
  -> controller validation
  -> service business rule
  -> repository persistence or provider call
  -> Prisma/PostgreSQL, object storage, or external provider
  -> standardized response or localized ProcureX API error

Public Content And Guest Marketplace

Frontend: publicBackend: /api/public + /api/procurement/publicDatabase: content + procurement

Key Features

The public layer introduces ProcureX before authentication. It includes welcome content, guest marketplace browsing, public contact entry, privacy policy, terms and conditions, and navigation into registration or sign-in. It lets suppliers discover opportunities and lets buyers understand the platform without entering protected workspaces.

Frontend Implementation

Routes such as /, /guest-marketplace, /about, /privacy, /terms, /contact, /help, and /status are declared as public routes. Public pages live under client/src/features/public, with ProcureX prototype-derived pages lazy-loaded through the page registry. Public API helpers fetch marketplace data, welcome landing data, page versions, and legal versions.

Backend Implementation

The public module exposes page and legal-version endpoints: GET /api/public/pages/:pageKey and GET /api/public/legal/current. The procurement module also exposes public welcome and marketplace data through /api/procurement/public/welcome and /api/procurement/marketplace. Public services provide stable fallback content when repository access is unavailable.

Database Design

Content is backed by PublicPageVersion, PublicPageKey, and PublicPageStatus. Marketplace visibility comes from procurement tables such as Tender, with fields for status, visibility, title, dates, buyer organization, and tender details.

Testing And Validation

Coverage includes public service tests for latest published page DTOs, fallback content, and current legal versions. Client tests verify public welcome rendering, public content pages, contact page behavior, marketplace rendering, and auth-route behavior that keeps help open publicly.

Security Notes

Public endpoints avoid session requirements, but still run through global Helmet, CORS, request body size, and error handling. Public legal content is read-only. Guest marketplace data is constrained to public or intended marketplace visibility instead of exposing private buyer drafts.

Organization And Access Capabilities

Frontend: identity/workspace/admin consumersBackend: /api/organizationDatabase: organization

Key Features

Organization modeling is the platform backbone for buyer and supplier behavior. Instead of assigning buyer or supplier as a login role, the platform attaches capabilities to the organization. This lets one company act as buyer, supplier, or both while the login account remains a normal user account.

Frontend Implementation

The user sees organization capability indirectly through workspace access, procurement actions, bidding actions, and account profile/verification screens. Route guards and session state consume capabilities returned by the identity access/session flow to decide whether core procurement routes can be entered.

Backend Implementation

The organization module exposes a status route and provides a stable boundary for future organization-specific APIs. Identity and procurement services use organization context heavily when creating verification profiles, assigning memberships, evaluating trust, creating tenders, saving bids, and applying RLS context.

Database Design

Core tables include Organization, OrganizationMember, OrganizationCapability, OrganizationProfile, BuyerProfile, SupplierProfile, TrustTierHistory, and PermissionOverride. Organization kind separates company organizations from platform administration organizations.

Testing And Validation

Identity production-auth tests validate computed IAM access context, dual buyer/supplier capabilities, local sanctions/watchlist routing, duplicate approved registry handling, and company verification outcomes. Route guard tests validate capability-sensitive access decisions for procurement core routes.

Security Notes

Organization id is part of the database context through withDbContext. RLS helper functions compare current organization ids to row ownership fields. Admin users can inspect platform data through admin context while normal users remain bounded by their own organization and workflow assignments.

Help Centre FAQ Assistant

Frontend: helpCentreBackend: /api/help-centreDatabase: audit only

Key Features

The Help Centre provides categories, FAQ lists, FAQ details, popular questions, category pages, suggestions, and a message endpoint that matches user questions to approved FAQ answers. It is intentionally guidance-only and warns users not to include secrets or confidential procurement data.

Frontend Implementation

The active IDE file client/src/features/helpCentre/api/index.ts defines helpCentreApi. It calls status, categories, FAQs, FAQ detail, popular, category, suggestions, and message endpoints. The API types come from @procurex/shared, which keeps FAQ DTOs consistent between client and server.

Backend Implementation

The server help-centre module loads categories and a catalog of FAQs, validates the complete FAQ knowledge base at service startup, normalizes question text, matches questions by FAQ content, user role, selected category, current path, account type, and capabilities, and returns either a matched response or a safe fallback response. The message endpoint accepts optional auth, so signed-in users can get context-aware help while public users still receive approved general guidance.

Database Design

The FAQ catalog itself is source-backed rather than modeled as a Prisma table in the current implementation. The module writes safe audit events to the compliance audit table for matched and unmatched FAQ interactions, storing metadata such as match state, faq id, category, confidence, path, account type, and capabilities without storing sensitive question content.

Testing And Validation

Help-centre service tests validate the complete FAQ knowledge base, exact and alternative bid submission matching, punctuation and case normalization, keyword and category relevance, weak match handling, matched and fallback responses, safe audit metadata, category lists, popular FAQs, category FAQs, suggestions, and individual FAQ lookups. The validation layer requires at least 250 enabled FAQs, non-empty titles, summaries, questions, keywords, steps, valid roles, valid categories, valid action paths, and valid related FAQ references.

Security Notes

The message endpoint has a dedicated in-memory rate limit keyed by hashed IP. Optional session lookup is tolerant of invalid tokens and does not make public help depend on login. Fallback warnings explicitly tell users not to include passwords, verification codes, bid prices, bank details, or confidential documents.

Security Coverage

HTTP And Browser Protections

The Express app uses Helmet with an explicit Content Security Policy. CSP directives restrict default sources, base URI, object sources, frame ancestors, script sources, connect sources, image sources, style sources, font sources, and Turnstile frames. CORS permits configured production origins or local development origins and rejects unapproved browser origins with a standardized ProcureX API error.

Authentication And Session Protections

Public auth endpoints are protected by auth rate limits and Turnstile checks. The client stores and injects bearer tokens through a shared API client. Route guards keep unauthenticated users away from protected pages, keep unverified users away from normal workspaces, and keep normal users away from admin pages.

Rate Limiting

General auth and mutation rate limits use express-rate-limit with Redis when configured and memory fallback outside production. Keys are hashed. Rate-limit events can be audited with hashed IP and user agent. Specific routes use profiles: public auth, sensitive identity, procurement contact verification, award sample mutation, post-award finance mutation, and Help Centre message rate limiting.

Production Configuration Enforcement

Production startup validation requires real CORS origins, Redis URL, Turnstile secret, public app URL, Resend configuration, SMS provider configuration, registry provider configuration, Mailboxlayer key, and signature hash secret. It rejects local-only providers such as dev-console notification delivery, SMTP identity email in production, and WhatsApp phone verification where marked local/testing-only.

Data Access And RLS

Database context utilities set transaction-local values for current user id, current organization id, current account type, and current capabilities. RLS SQL defines helper functions such as current user, current organization, current account type, current capabilities, admin detection, same organization checks, and capability checks. Policies protect users, accounts, sessions, verification profiles, tenders, bid rows, evaluation scores, admin datastore rows, and compliance rules.

Validation And Error Safety

Controllers use validators before business logic. Central error handling converts Zod issues into a stable error response with field errors. Internal server errors are kept generic so raw implementation messages, database messages, and secrets do not leak to users. Client notification tests confirm technical text stays hidden from ordinary notification summaries.

Signing And Audit

Signing credentials and signed actions give high-impact workflow decisions a cryptographic/audit boundary. Keyphrase-protected actions include identity verification submission, award approvals, supplier award responses where needed, contract signing, payment approvals, termination decisions, and closeout decisions. Audit payload helpers record useful metadata while avoiding raw secrets.
