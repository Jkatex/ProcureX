1. Layer Architecture

ProcureX is implemented as a monorepo with separate client, server, shared contract, documentation, infrastructure, and future intelligence service boundaries.

React Client

  • React 19 and Vite.
  • React Router selects screen surfaces.
  • Feature code under client/src/features.

Route Guards

  • Public routes render directly.
  • Protected routes require session.
  • Verified routes require trusted organization context.
  • Admin routes require admin account.

Express API

  • Helmet, CORS, JSON body limits, health endpoint.
  • Modules mounted under /api/*.
  • Central 404 and error response handling.

Domain Services

  • Routes call controllers.
  • Controllers validate transport concerns.
  • Services orchestrate business behavior.
  • Repositories own persistence access.

Data Services

  • Prisma and PostgreSQL are model truth.
  • Redis, Elasticsearch, and MinIO support local infrastructure.
  • RLS helpers prepare organization-aware access.
client route -> route guard -> API adapter -> Express router -> controller -> service -> repository -> Prisma -> PostgreSQL

6. Frontend Deep Dive

The frontend is the user's operating surface. It does not own procurement truth by itself; it presents routes, gathers input, calls APIs, renders returned state, and protects routes from users who do not yet have the required session, verification, or admin context.

Frontend Area Responsibility Important Behavior Risk If Broken
Application entry Bootstraps React, providers, routing, localization, theme, and shared state. The entrypoint should create one coherent app shell and let routing decide which feature page is visible. Users see blank pages, duplicated providers, broken theme, or incorrect localization.
Router Maps browser paths to public, protected, verified, and admin screens. Routes such as /dashboard, /bidding, /evaluation, and /admin should be guarded at the router layer instead of relying on each page to self-protect. Sensitive screens become reachable without the right session or trusted organization state.
Page registry Connects stable ProcureX page keys to React components derived from or aligned with the prototype. The registry helps keep route definitions compact and allows UI parity workflows to reason about page names consistently. Screenshot parity, navigation, or route mapping becomes hard to maintain.
Route guards Protect authenticated routes and redirect users based on session, verification, trust, permission, gate, or admin status. Guards represent product rules such as admin redirect, verified workspace requirement, and support paths for unverified users. Users enter the wrong workspace, become stuck after sign-in, or bypass trust gates.
Feature API helpers Wrap HTTP calls for identity, procurement, bidding, evaluation, awards, communication, and other features. API helpers hide transport details from pages and keep endpoint paths in the feature boundary. Pages duplicate request logic, error handling becomes inconsistent, and route changes become expensive.
Feature state and hooks Own local or shared UI state for each domain. Hooks and slices should present a clean interface to pages: load data, submit action, expose status, and show errors. Pages become difficult to test and business state leaks across feature boundaries.
Prototype compatibility Preserves visual and wording parity with the original static ProcureX reference. Styles and generated/maintained components should keep the production UI recognizable while allowing real routing and data. The production app drifts from approved stakeholder expectations or screenshot references. 
browser path -> router.tsx -> route guard -> Procurex page registry -> feature page -> hook/API helper -> Express API

7. Backend Deep Dive

The backend is organized around modules so each procurement domain can evolve without one large controller owning the entire platform. The shape is intentionally repetitive: repetition here is a maintainability feature because a developer can open any module and quickly find routes, controllers, services, repositories, validators, and local types.

Backend Piece Primary Job Deep Detail Good Change Pattern
app.ts Create the Express app. Configures security middleware, CORS, JSON parsing, health response, module mounting, 404 response, and centralized error handler. It should stay focused on application wiring, not domain behavior. Add cross-cutting middleware here only when it applies to the full API surface.
modules/index.ts Register module routers and base paths. Defines the API boundary list returned by /health. Adding a module should be explicit so operational tooling and tests can see it. Add a module definition, base path, and router factory together.
routes.ts Declare HTTP endpoints. Routes should remain thin. They should connect methods and paths to controller handlers and attach route-specific middleware where appropriate. Add endpoint paths here, then delegate all meaningful work to controllers and services.
controller.ts Handle HTTP request/response concerns. Controllers translate HTTP inputs to service calls and service results to HTTP responses. They should not become persistence layers or business policy engines. Parse params/body/query, call validator/service, return status and JSON.
validators.ts Validate external input. Validation should fail early and clearly before service logic assumes a stable shape. Zod schemas or local validators protect services from malformed transport data. Define schemas close to the module and keep error messages usable.
service.ts Own business behavior and orchestration. Services decide what a user action means in the domain: start registration, publish tender, submit bid, score evaluation, recommend award, create contract, record payment, or open compliance action. Put workflow decisions, domain transitions, and cross-repository coordination here.
repository.ts Own persistence access. Repositories isolate Prisma queries, transactions, and database-specific shape. They should not return raw persistence complexity to controllers. Add focused read/write functions and keep query ownership inside the module.
types.ts Define module-local contracts. Types define service inputs, outputs, module metadata, statuses, and local vocabulary that should not be duplicated across files. Update types when behavior or DTO shape changes, and move cross-workspace vocabulary to shared contracts when needed.
Lifecycle Domain Primary API Modules Typical Request Service Decision
Registration and trust /api/identity, /api/organization Start registration, verify OTP, activate email, set password, submit verification, admin decision. Whether a user can become active, trusted, and attached to a company context.
Tendering /api/procurement, /api/documents Create or update tender, attach documents, publish tender, inspect tender details. Whether a tender is complete, owned by the correct buyer organization, and visible to suppliers.
Bidding /api/bidding Create bid draft, update response, upload document reference, submit bid, issue receipt. Whether supplier organization can submit, whether deadline/status allows submission, and what version is final.
Evaluation /api/evaluation Load workspace, assign evaluators, save criterion score, advance evaluation stage. Whether the actor has workflow assignment and whether the scoring action is valid for the stage.
Award and contract /api/award-contract, /api/post-award, /api/financial Create recommendation, approve award, record response, draft/sign contract, create invoice or payment record. Whether evaluation evidence supports award, whether approval is complete, and whether execution records match contract state.
Governance /api/compliance-admin, /api/records, /api/communication, /api/dashboard Review compliance, search, inspect audit, send communication, load dashboard summary. Whether the actor has admin or workflow context and how the record should be presented for traceability.

8. Storage And Documents

Procurement work depends heavily on documents: tender packages, eligibility files, bid attachments, official generated documents, contract versions, signatures, evidence, invoices, receipts, inspection records, and closeout packages. The system therefore needs a clear distinction between structured database records and binary or generated document content.

Document Type Structured Record Storage Concern Traceability Concern
Profile image User profile or identity data references image metadata. Profile image storage validates allowed image data, size, and generated storage key behavior. Profile changes should remain attributable to the account and not expose unsafe arbitrary content.
Tender document TenderDocument and related tender fields. Files or generated HTML/PDF output must remain tied to the tender version or publication state. Suppliers and auditors need to know which document defined the opportunity.
Bid attachment BidDocument, BidVersion, and BidResponse. Documents must be associated with the correct supplier organization and bid version. Evaluation should use the submitted evidence, not an overwritten draft after deadline.
Official procurement document DocumentObject and official document module outputs. Generated documents should have stable source data, rendering logic, and object references. Generated output must be reproducible or at least explainable from stored source fields.
Contract version ContractVersion, clauses, parties, signatures, and negotiation records. Each version should preserve the agreed text or generated package that was reviewed. Parties need proof of exactly which contract was signed or negotiated.
Post-award evidence Goods receipt, inspection, progress report, service report, invoice, payment certificate, claim, or defect record. Evidence may include uploaded files, generated reports, photos, invoices, certificates, or inspection attachments. Payment, acceptance, penalties, claims, and closeout all depend on durable evidence. 
user upload or generated output -> validation -> document metadata -> object key/reference -> domain record -> audit/record timeline
Document design principle: a file alone is not enough. A procurement document becomes useful only when it is connected to a domain record, owner organization, lifecycle stage, timestamp, and audit trail.

9. Testing And Verification

Technical documentation should explain how the implementation proves it still behaves as described. ProcureX uses targeted server tests, client tests, route guard tests, service tests, seed tests, screenshot parity tools, Prisma validation, and row-level security verification to protect different layers of the system.

Test Layer What It Protects Example Risk Expected Verification
Module validation tests Backend module registry and consistent module definitions. A module is created but not mounted, or health output no longer reflects API boundaries. Tests confirm module keys, base paths, and app behavior.
Identity tests Registration, production auth safety, delivery providers, signing, profile image behavior, and verification decisions. Development bypass leaks into production, invalid image data is accepted, or account flow grants access too early. Focused identity tests cover sensitive account and trust behavior.
Procurement and bidding tests Tender creation, marketplace behavior, bid submission rules, repository logic, and seeded demo workflows. Supplier sees buyer-only data, tender ownership is wrong, or bid version rules regress. Service/repository tests validate ownership and workflow state.
Evaluation tests Evaluation intake, criteria, scoring, assignments, demo seed readiness, and award handoff. Evaluation cannot load realistic seeded data or scores no longer produce recommendation-ready state. Evaluation service and seed tests protect the end-to-end handoff.
Awards and post-award tests Award-contract workflows, role access, contract detail, mobile/desktop views, and post-award app behavior. Award action appears to the wrong actor, contract view breaks, or post-award state cannot be navigated. Flow tests and component tests verify user-visible behavior.
Route guard tests Public, protected, verified, admin, support, and redirect behavior. Unverified users access core workspace or admins land on buyer/supplier routes. Client tests assert routing outcomes under mocked session states.
UI parity screenshots Visual alignment between static prototype and production React routes. Production screens drift from approved reference UI during implementation. Screenshot capture compares reference and target pages for desktop and mobile.
Prisma validation and RLS verification Database schema integrity and organization-aware access assumptions. Schema change breaks generated client or data isolation rules become unreliable. db:validate and verify:rls confirm database-level expectations.
Acceptance Area Deep Acceptance Criteria
Route correctness Every user-facing route in the journey docs is public, protected, verified, or admin-only according to the router and product rules.
Module correctness Every API module named in documentation is registered in the backend module list or clearly described as an expansion boundary.
Data correctness Every major lifecycle record maps to an actual Prisma model group: identity, organization, tender, bid, evaluation, award, contract, post-award, document, communication, record, or audit.
Security correctness Documentation must preserve the rule that account type is not procurement capability, and that sensitive work is organization-aware.
Operational correctness Commands and file references should match workspace scripts and source paths so another engineer can validate the system locally.

10. Operations

Concern Command Or File Purpose
Health check GET /health Returns service status and registered API modules.
Local infrastructure npm run infra:up Starts PostgreSQL, Redis, Elasticsearch, and MinIO with Docker Compose.
Database validation npm run db:validate Validates the Prisma schema.
Database migrations npm run db:migrate Applies local migrations for development.
Build order npm run build Builds shared contracts, then server, then client.
Verification npm test and verify:rls Runs server/client tests and database security verification where configured.
Maintenance rule: add new behavior inside the domain boundary that owns it. Keep HTTP handling in controllers, business decisions in services, persistence in repositories, shared vocabulary in shared contracts, and model truth in Prisma.
